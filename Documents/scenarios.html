Scenarios:

  1. Name: Creating a New Team
    Description:

    The process starts when the user presses the "Create Team" button. They will be prompted to fill fields relating to the team's name and purpose. Once the user submits this form, the application will ensure all necessary fields are filled and create new instance of a team within the database; if the form is not filled, it will instead display an error message stating what is missing. After team creation, the user is assigned the role of project manager and is presented with the option to invite new team members and create tasks.


  2. Name: Creating and Assigning Tasks
    Description:

    Any user who is designated as the project manager of the relevant project, or another role granted sufficient permissions, is able to see the "Create Task" button. Upon button press a form is presented with the fields necessary for task creation: task name, description, priority level, completion state, and initial team members. The task name is the only field that is an absolute requirment. When the user submits the form, if the task name field is not blank. a new task is inserted into their database and will now appear in the dashboard of their project. After the task has been created, user's with valid authority will be able to view and click an options button within the task. They will be able to assign the task to more users via a text field within this menu.


  3. Name: Updating Task Progress
    Description:

    Any user with relevant authority will be able to see the "options" button for a task from the dashboard. Clicking on this button will open a menu containing all the information pertaining to the task that the user is able to manipulate. The project manager and users granted the authority to do so can edit all facets of a task, while any user who is assigned to the task will be able to at least view edit the task progress unless this authority has been stripped via a special role. By changing which radio button is selected and submitting the form, the user will be able to update the task's progress. This has the potential to fail if the user has the ability to edit multiple fields and provides an invalid entry to one of them.


  4. Name: Team Communication Using Discussion Boards
    Description:

    Several team members find a design issue that affects multiple tasks and multiple roles. One contributor goes ahead and creates a post on the common discussion board explaining the problem and asking for solutions. Other team members then reply with feedback and alternative approaches, allowing the team to plan a solution accordingly. For more sensitive topics, the project manager creates a private discussion board, granting access only users with relevant roles to view the post. If a user without permission attempts to view the private board, the system blocks their access and returns them to the dashboard. At the end of discussion, messages can be locked by the project manager, and the system stores the messages for later reference.


  5. Name: Searching and Filtering Tasks
    Description:

    A team member wants to locate all high-priority tasks that are incomplete. They use the search bar on the dashboard and apply filters for task status, priority, and assignment, which the system returns a list of the tasks that match the search criteria. However, another team member decides to search for low-priority, optional tasks. However, no tasks match their desired priority level, so the system informs them no matching results were found and allows them to adjust their filters. Once the system locates the desired tasks, the team member can open them directly from the filtered list.


  6. Name: Viewing Progress and Deadlines
    Description:

    The project manager opens the dashboard to check the team's progress and the total completion of the project. A weighted progress bar shows the percentage of completed tasks, and the calendar highlights upcoming deadlines for future tasks. Using this information, the project manager decides to reassign responsibilities of users and/or change the priorities of tasks, adjusting the requirements of the project on the fly as progress is made.


  7. Name: Receiving Deadline Notifications
    Description:

    When a deadline for an incomplete task is approaching, system automatically sends a browser notification to all members assigned to that task. Task name and due tame is mentoned in the notification, suggesting the user to review and update their progress. If browser notifications are disabled no reminder will be delivered.


  8. Name: Role Management and Access Control
    Description:

    A project manager navigates to the team management section and assigns a role to the team member, which immediately updates member`s access right within the project. If a role needs to be changed/revoked, the project manager can do it from the same interface and it will cause the team member to lose access to certain parts of the project.


  9. Name: Organizing Tasks into Folders
    Description:

    A project manager creates a folder and, if needed, restricts access to it for certain roles, and moves tasks into it. Team members that dont have a required role will not be able to see the folder or its contents. Individual acces can be granted or revoked by the project manager.
